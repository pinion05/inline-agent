# 문맥 프록시 Wayfinder 설계

## 목적

문맥·트레이토리·Dashboard 관련 이슈를 하나의 결정 지도로 정리한다. 지도의 종료 조건은 다음 구조로 전환하기 전에 필요한 설계 결정, 기존 이슈의 병합·폐기 판단, 구현 순서가 모두 명확해지는 것이다.

실제 구현은 이 지도에서 모든 결정 항목이 해결된 후 별도의 구현 계획으로 진행한다.

## 범위

포함:

- 세션 작업 기록의 영구 보존과 재개
- 문맥 프록시
- 실제 도구 결과 단위의 원문 보존과 축약
- API 입력 토큰 관리
- 수동 문맥 줄이기
- 이해 토큰, 엔드턴 요약, 후방 삽입
- 실제 HTTP 요청 기반 Dashboard

제외:

- Codex 인증
- skills 자동감지
- API 재시도 정책 자체
- 파일 시스템 스냅샷
- 일반 TUI 디자인

## 확정된 원칙

### 1. 디스크 세션 기록이 유일한 원본이다

세션에서 발생한 메시지는 트리 준비형 JSONL 파일에 뒤로만 추가한다. 기존 항목을 압축본으로 교환하거나 삭제하지 않는다.

각 기록은 현재 OpenAI 호환 `Message`를 그대로 감싸는 최소 정보만 가진다.

```json
{
  "type": "message",
  "id": "a1b2c3d4",
  "parentId": "prev1234",
  "timestamp": "2026-07-16T00:00:00.000Z",
  "message": {
    "role": "tool",
    "tool_call_id": "call_1",
    "content": "..."
  }
}
```

`id`와 `parentId`는 현재 직선형 대화에서도 기록한다. 이후 `/resume`, 되돌리기, 분기를 기존 기록 수정 없이 구현할 수 있어야 한다.

### 2. 세션 저장소가 저장과 현재 경로 선택을 담당한다

세션 저장소의 책임:

- JSONL 생성·열기·검증
- 메시지 항목을 디스크에 먼저 추가
- 메모리 항목 목록과 ID 위치표 갱신
- 현재 끝 지점 관리
- 현재 끝 지점에서 루트까지의 경로를 직선형 `Message[]`로 제공

디스크 쓰기에 실패하면 메모리 상태도 전진하지 않는다.

### 3. 문맥 프록시는 상태를 저장하지 않는다

문맥 프록시는 세션 저장소에서 현재 경로를 읽어 이번 API 호출에 보낼 새 `Message[]`를 만든다. 자체 메시지 배열, 축약 상태, 마지막 요청 복사본, Dashboard 상태를 보관하지 않는다.

모든 세션 메시지 추가와 API용 메시지 읽기는 문맥 프록시 경계를 통과한다. 외부 모듈은 세션 저장소의 내부 배열 참조를 받지 않는다.

### 4. 문맥 절감은 API 전송 복사본에서만 일어난다

문맥 프록시는 매 API 요청마다 정해진 규칙을 처음부터 적용한다. 결과는 요청 후 폐기한다.

- 세션 기록은 변경하지 않는다.
- 최신 사용자 메시지 직전의 일회성 시스템 정책을 삽입한다.
- 입력 토큰 한도에 맞게 도구 결과를 줄인다.
- 설정 변경은 같은 도구 실행 반복의 다음 API 요청부터 반영한다.
- 초기 구현은 매번 전체 경로를 다시 계산한다. 별도 재사용본은 성능 측정 전까지 도입하지 않는다.

### 5. 도구 결과는 실제 호출별로 처리한다

에이전트의 `tool_calls[]` 묶음은 원래 순서와 호출 ID 연결을 유지한다. 각 도구 결과는 독립적으로 원문 보존 또는 축약 대상으로 선택한다.

예를 들어 호출 세 개 중 최근 결과 두 개를 원문으로 보존하면 API 요청은 다음 의미를 유지한다.

```text
assistant(call1, call2, call3)
tool(call1, 축약 결과)
tool(call2, 원문 결과)
tool(call3, 원문 결과)
```

설정값은 에이전트 응답 묶음 수가 아니라 실제 원문 도구 결과 수를 뜻한다.

### 6. Dashboard의 유일한 원천은 실제 HTTP 요청이다

Dashboard는 세션 기록이나 문맥 프록시 결과를 다시 해석해 요청을 추정하지 않는다. OpenAI SDK가 직렬화한 뒤 실제 `fetch`에 넘기는 HTTP 요청 본문을 감시 지점에서 복제한다.

- 모든 최초 요청과 재시도를 시간순으로 기록한다.
- 기본 화면은 가장 최근 전송 시도를 표시한다.
- 인증 헤더는 수집하지 않는다.
- 실제 본문 문자열과 해석 가능한 경우 JSON 값을 함께 보존한다.
- 토큰 및 메시지 표시는 감시한 요청 본문을 기준으로 계산한다.

## 전체 데이터 흐름

```text
사용자·에이전트·도구 메시지
          ↓
문맥 프록시의 추가 기능
          ↓
세션 저장소
├── 트리 준비형 JSONL에 추가
└── 현재 대화 경로 제공
          ↓
문맥 프록시의 API 메시지 구성
├── 실제 도구 결과 단위 선택
├── 요청 복사본에서만 축약
├── 일회성 정책 삽입
└── 입력 토큰 한도 검사
          ↓
OpenAI SDK
          ↓
실제 HTTP 요청 감시기
├── Dashboard 기록
└── LLM 서버 전송
```

## Pi에서 참고할 구조

Pi는 세션 항목을 JSONL에 뒤로만 추가하고 메모리 ID 위치표를 갱신한다. 각 항목의 `id`와 `parentId`로 현재 대화 경로를 복원한다. 문맥 생성은 저장된 전체 기록을 삭제하지 않고 현재 경로와 문맥 선택 규칙으로 별도 메시지 목록을 만든다.

inline-agent는 Pi의 전체 메시지 형식이나 기능을 복제하지 않는다. 다음 원칙만 참고한다.

- 디스크 우선 추가
- 뒤로만 추가되는 JSONL
- ID 기반 현재 경로
- 저장된 전체 기록과 LLM 전송 문맥 분리
- 문맥 줄이기를 삭제가 아닌 별도 기록과 선택으로 표현

## Wayfinder 결정 항목

즉시 검토 가능한 결정 항목:

1. 세션 기록 형식과 안전한 쓰기 규칙
2. 문맥 프록시의 도구 결과 축약 규칙
3. 실제 HTTP 요청 감시의 보안과 실패 규칙
4. 기존 문맥 관련 이슈의 현행 상태 판정

위 결정에 의존하는 항목:

5. `/resume`이 복원해야 하는 상태 범위
6. `/compact`가 원본 삭제 없이 동작하는 의미
7. 이해 토큰·엔드턴 요약·후방 삽입의 통합 방향
8. 전체 전환 순서와 기존 이슈 정리 순서

## 아직 세부 항목으로 만들지 않는 영역

다음 영역은 선행 결정 결과가 나올 때까지 지도의 미지정 영역으로 둔다.

- 세션 분기와 되돌리기의 사용자 화면
- 장기간 쌓인 세션 파일 정리 정책
- 이해 토큰 품질 평가 방법
- 세션 간 지식 추출
- 디스크 원문 로그를 LLM이 다시 읽는 정확한 방식

## 검증 원칙

구현 계획은 최소한 다음 증거를 요구해야 한다.

- 프로세스를 종료하고 다시 열어도 세션 메시지와 현재 경로가 동일함
- 디스크 쓰기 실패 시 메모리 기록이 전진하지 않음
- 문맥 프록시 호출 전후로 저장된 기록의 바이트와 객체 값이 바뀌지 않음
- 호출 세 개와 원문 보존 설정 두 개에서 실제 원문 도구 결과가 정확히 두 개임
- 에이전트 호출 묶음과 각 `tool_call_id` 연결이 API 요청에서 유효함
- Dashboard 기록의 본문 문자열이 실제 `fetch`에 전달된 본문과 동일함
- 최초 요청과 모든 재시도가 별도 전송 시도로 표시됨
- 인증 헤더와 API 키가 Dashboard 기록에 포함되지 않음
